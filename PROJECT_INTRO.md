# 剧本评测系统介绍

### 1. 项目目标

评测对象：“8-12集精短悬疑电视剧”
理由：不同剧集类型的质量标准差异大，悬疑剧天然适合信息释放、伏笔回收、反转公平性等可证据化判断，因此本次搭建优先从该类型切入


### 2. 评测思路

┌────────────────────┐
│  原始剧本 / 分集大纲 │
└─────────┬──────────┘
          ↓
┌────────────────────┐
│ Node 0 输入校验      │
│ 判断材料是否能评     │
└─────────┬──────────┘
          │
#系统首先要确认用户提供的是完整剧本、分集大纲、人物小传，还是只有故事梗概。不同材料支持的评估深度不同。
例如，只有故事梗概时，可以粗评结构和核心设定，但不能精确判断场景功能、人物对白和具体节奏。
          │
          ↓
┌────────────────────┐
│ Node 1 剧本切分      │
│ 拆成集 / 场 / 事件   │
└─────────┬──────────┘
          │
          ↓
┌──────────────────────────────┐
│ Stage 1 分集事实抽取          │
│ 按集抽取结构、人物、信息、伏笔 │
└─────────┬────────────────────┘
          │
按集抽取更稳定，也方便失败后单集重跑
          │
          ↓
┌──────────────────────────────┐
│ Stage 2 全局事实汇总          │
│ 合并成人物线、信息线、伏笔线   │
└─────────┬────────────────────┘
          ↓
┌────────────────────┐
│ Node 3 问题候选生成 │
│ 标记疑似风险点       │
└─────────┬──────────┘
          ↓
┌──────────────────────────────┐

└─────────┬────────────────────┘

│ 输出分数、问题、建议 │

当前评测维度主要包括：

- D1 悬疑信息控制
- D2 人物行动链
- D3 分集结构与节奏推进
- D4 情节因果与逻辑可信度
- D5 场景戏剧张力与有效密度
- D6 主题表达与现实质感
- D7 结尾回收与整体完成度

其中 D1-D4 是核心维度，权重更高，因为悬疑剧最关键的是信息控制、人物行动、结构推进和逻辑可信度。如果这些基础不稳，即使题材新颖或表达有亮点，整体开发风险也会比较高。

七个维度为 0-5 分制，再按权重换算成 100 分制总分：

- 5.0：接近播出级成熟稿。
- 4.5-4.9：成熟可开发。
- 4.0-4.4：整体成立，但仍有局部风险。
- 3.5-3.9：成立但有明确修改点。
- 3.0-3.4：勉强可读，核心机制有短板。
- 2.0-2.9：关键机制不稳，需要重点修改。
- 1.0-1.9：大面积失效，需要重构。
- 0-0.9：当前维度基本不成立或无法评价。

任一维度低于 3.0，会触发低分专项诊断。诊断需要具体指出问题维度、分数、严重程度、核心问题、位置、证据、影响和可执行修改建议。

同时，评分后还会做校准。系统会生成 Round A 和 Round B 两轮评分，再比较差异。0-5 分制下，10% 稳定性阈值是 0.5 分；如果同一维度两轮差异超过 0.5，校准节点必须标记为“需仲裁”，回到证据和硬性降分规则，原则上采用更保守的分数，而不是简单取平均。

### 3. 文件架构

当前项目的核心目录如下：

```text
input/
  wuxian_episode_script.md（《雾线》）
  lixian_zhengren_episode_script.md（《离线证人》）

prompts/
  00_input_validation.md
  01_script_split.md
  02_episode_structure_extract.md
  03_scene_function_extract.md
  04_character_action_extract.md
  05_information_release_extract.md
  06_setup_candidate_extract.md
  07_global_reduce.md
  08_issue_candidates.md
  score_D1_suspense_information.md
  score_D2_character_chain.md
  score_D3_episode_structure.md
  score_D4_logic.md
  score_D5_scene_density.md
  score_D6_theme.md
  score_D7_ending.md
  score_calibration.md
  low_score_diagnosis.md
  final_report.md

src/script_quality_evaluator/
  cli.py
  pipeline.py
  llm_client.py
  report.py
  scoring.py
  extraction.py
  schemas.py
  io_utils.py

output/
  每篇剧本一个独立文件夹/
    final_report.md
    final_score.json
    low_score_diagnoses.json
    progress/
      run_manifest.json
      checkpoints/
      episode_map/
      中间抽取和评分结果
```

其中：

- `input/` 放待评测剧本，目前有两篇测试版剧本（GPT生成）
- `prompts/` 是各节点的模型提示词。
- `src/script_quality_evaluator/` 是程序主逻辑。
- `output/` 是输出目录，每篇剧本会生成独立文件夹，避免多篇测试结果混在一起。
- `final_report.md` 是最核心的阅读产物。
- `progress/` 用于监督过程，比如模型如何切分、如何抽取人物行动、如何给分。
- `run_manifest.json` 会记录输入文件和内容指纹，避免 resume 时误复用另一篇剧本的 checkpoint。

### 4. 当前能力

目前系统已经具备 MVP 基础能力：

- 支持一键运行默认剧本，并在命令行中实时显示生成进度。
- 支持中断后 `--resume`，复用已经完成的节点。
- 支持每篇剧本独立输出，避免文件混乱。
- 支持便宜模型做切分和抽取，较强模型做评分和报告。
- 支持 `--rerun-judging`，复用前面的结构化抽取结果，只重新跑评分和最终报告。
- 支持两轮评分稳定性检查：同一维度 Round A / Round B 差异超过 0.5 时进入仲裁。
- 支持 0-5 分内部评分，并换算为 100 分制总分，兼顾规则清晰度和对外可读性。
- 支持离线规则模式，用来检查程序链路是否能跑通。

### 5. 当前限制

这个系统目前还不是一个最终成熟的自动评审产品，主要限制包括：

- 成本有限，目前主要使用 DeepSeek 做测试，还没有系统比较不同模型在剧本评估任务上的表现。
- 当前测试样本篇幅较短，而且有 AI 生成剧本，和真实项目的复杂度还有差距。
- 真实剧本可能会有更复杂的格式，比如对白剧本、场景剧本、分集大纲混排、多版本修订稿等，需要继续增强切分稳定性。
- 评分质量还依赖 prompt，后续需要用更多真实样本校准分数尺度，避免过松或过严。
- 问题定位目前可以给到集数、场景、人物、线索层面，但还需要进一步提高“问题类型”和“修改建议”的准确性。
- 对类型差异的适配还不够充分。悬疑、刑侦、家庭伦理、科幻悬疑、现实题材悬疑可能需要不同评分侧重。

### 6. 后续迭代方向

后续会优先从以下几个方向优化：

1. 增加真实剧本样本测试  
   用真实项目中的完整剧本、分集大纲和不同成熟度文本来测试，观察模型在哪些节点容易误判。

2. 持续迭代 prompt  
   重点优化剧本切分、人物行动链、信息释放、问题候选和评分节点，让输出更稳定、更少臆测。

3. 建立评分标尺  
   通过人工标注或专家样例建立 0-5 分参照，让模型知道什么是 2 分、3 分、4 分、5 分，并持续检查总分换算是否符合开发评审直觉。

4. 强化问题定位  
   让报告不只说“人物动机不足”，而是能指出具体是哪一集、哪一场、哪一个行动缺少前置动机，以及建议补哪类场景。

5. 支持多模型分工  
   继续保留低成本模型做数据清洗和结构化抽取；在评分、校准、最终报告等更依赖理解力的节点，切换更强模型。

6. 增加人工复核入口  
   后续可以让使用者只修改中间 JSON 或补充人工判断，然后复用已有抽取结果重新评分。

7. 优化最终报告表达  
   让报告更像专业开发评审稿，而不是模型总结稿：先给结论，再给证据，再给修改优先级。

8. 加入回归测试集  
   保留一批典型剧本样本，每次改 prompt 后都跑一遍，检查分数是否明显漂移、问题定位是否退化。

### 7. 一句话总结

当前版本已经能跑通完整链路，但真正的质量提升还需要在真实剧本测试中持续校准 prompt、评分尺度和问题定位能力。
